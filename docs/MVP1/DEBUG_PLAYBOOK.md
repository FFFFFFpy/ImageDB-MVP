# MVP1 Debug / 实战测试手册

## 1. 当前阶段

MVP1 已定性为功能完成，当前进入：

```text
Debug / 实战测试 / 发布验收补强
```

本阶段目标不是继续扩功能，而是确认：

- 真实数据是否能稳定导入。
- 错误路径是否可诊断。
- 中断 / 重启 / 取消 / 恢复是否可靠。
- 大目录和长时间运行是否稳定。
- 安装包在 clean Windows 上是否独立可用。

## 2. 实战测试建议顺序

### 第一轮：小规模真实图集

目标：确认真实图片格式和路径没有基础问题。

建议规模：

- 5–20 个图集。
- 100–500 张图片。
- 包含中文、空格、括号、长文件名。
- 包含 jpg / png / webp。

观察点：

- 托管 PostgreSQL 是否稳定启动。
- 源目录是否正确识别一级图集。
- 扫描进度是否准确。
- 审核页预览是否正常。
- frozen plan 的保留 / 排除数量是否符合预期。
- 入库后的目标目录结构是否正确。
- 源图集归档是否符合预期。

### 第二轮：重复和覆盖图集

目标：确认重复检测和图集级规划不会误删、误留。

建议构造：

- 同图集内完全重复图片。
- 跨图集完全重复图片。
- 一个图集完整复制另一个图集。
- 一个图集是另一个图集的子集。
- 一个图集包含另一个图集的全部图片并额外多几张。

观察点：

- 自动重复是否排除正确。
- 完整覆盖图集是否作为 skipped album。
- 超集图集是否被保留。
- frozen plan 和 Commit 结果是否一致。

### 第三轮：异常文件

目标：确认坏文件不会拖死整个导入。

建议构造：

- 空文件。
- 损坏图片。
- 后缀是图片但内容不是图片。
- 超大图片。
- 无权限文件。

观察点：

- 错误是否进入错误列表。
- 其它正常图片是否继续处理。
- 错误信息是否能定位文件。
- Commit 前是否不会把失败图片误入库。

### 第四轮：中断恢复

目标：确认用户最像用户的时候，软件不会像软件一样崩。

建议操作：

- 扫描中取消。
- Commit 中取消。
- Commit 中强杀进程。
- Commit 后重启应用。
- 断开 mounted 目录后恢复。
- 源目录在分析后被改动。

观察点：

- Recovery 页是否能发现事务。
- Reverify 是否能给出明确结果。
- 重复执行是否不产生重复入库。
- 数据库记录和文件系统是否一致。
- 源图集是否在成功确认前保持完整。

### 第五轮：大目录 / 稳定性

目标：进入发布硬化。

建议规模：

- 1k 图片。
- 10k 图片。
- 100k 图片，作为后续重压，不必一开始就折磨自己，虽然人类很擅长。

观察点：

- 扫描耗时。
- Commit 耗时。
- 内存增长。
- UI 卡顿。
- PostgreSQL 日志。
- 是否有长时间无进度。
- 取消响应是否及时。


### 清空历史数据库并重新开始

当历史数据库中的旧记录不再需要、希望回到可直接导入的空 ImageDB 时：

1. 打开“设置 → 危险操作 → 清空历史数据库”。
2. 确认当前没有扫描、提交、恢复或外部数据库迁移任务。
3. 输入 `从零开始`，再点击“清空数据库并重新开始”。

该操作不会迁移或保留旧数据。它会清空 ImageDB 自己的导入记录、审核结果、图库数据库记录和事务记录，并初始化当前版本所需的空表结构；数据库连接配置、应用设置、图库目录及其中的实际文件不会被删除。若仍有未完成、冲突或待清理的文件事务，或存在尚未过期的图库根目录租约，后端会拒绝清空，以免丢失恢复证据。

重置、数据库 schema 初始化使用 PostgreSQL 级独占 advisory lock；扫描、提交、恢复在各自数据库连接存续期间持有同一协议的共享 advisory lock。因此，同一数据库上的其他 ImageDB 实例正在执行关键任务时，重置会立即拒绝，且重置期间其他实例不能启动这些任务。进程内 `CriticalOperationGuard` 仍用于当前实例的界面状态和任务互斥，但不再承担跨实例安全边界。

外部 PostgreSQL 中不属于 ImageDB 的表不会被主动删除；如果外部对象依赖 ImageDB 表，清空会失败并整体回滚。

### 完整发布签字

```bash
pnpm release:gate
```

## 4. Debug 记录模板

每个实战问题建议按以下格式记录，别写“好像坏了”这种对未来自己施法的句子。

```md
## 问题标题

- 日期：YYYY-MM-DD
- 环境：Windows / macOS / NAS / 外接盘 / 本地盘
- 分支 / 提交：main @ <sha>
- 数据规模：图集数 / 图片数 / 总大小
- 数据类型：jpg/png/webp/中文路径/长路径/坏图等
- 操作流程：
  1. ...
  2. ...
  3. ...
- 期望结果：
- 实际结果：
- 是否可复现：是 / 否 / 偶发
- 日志位置：
- 诊断包：
- 初步判断：
- 处理状态：未处理 / 修复中 / 已修复 / 暂缓
```

## 5. 问题优先级

### P0：必须立即修

- 数据丢失。
- 源图集在未成功入库前被破坏。
- 数据库记录和文件系统明显不一致。
- Commit 后无法恢复。
- 应用无法启动。
- 托管 PostgreSQL 初始化失败且无法自恢复。
- clean Windows 安装包无法启动。

### P1：Debug 阶段应修

- 大目录性能严重不达标。
- 扫描 / Commit 进度卡住但无明确错误。
- Recovery 诊断不清楚。
- 审核页预览失败但无定位信息。
- 错误文件影响正常文件导入。
- 长路径 / Unicode / 空格路径问题。

### P2：可排期

- 文案不够好。
- UI 小体验问题。
- 低频边界格式。
- 性能优化但不阻断。
- 更多诊断信息。

### P3：不要塞进 MVP1

- 新算法。
- 云同步。
- 多用户。
- 大 UI 改版。
- 非必要架构重构。

## 6. 日志和诊断

优先收集：

- 应用日志。
- PostgreSQL 日志。
- 当前 `settings`。
- import run id。
- transaction id。
- 源目录结构摘要。
- 目标图库目录结构摘要。
- `RELEASE_DOD.md` 中对应 gate 的状态。

不要上传用户真实图片内容到问题记录。需要复现时，用最小复现数据集。

## 7. 修复准入

每个修复至少说明：

- 它修的是哪个真实问题。
- 影响 scan / review / commit / recovery 哪一段。
- 是否会改变 frozen plan 结果。
- 是否需要 migration。
- 是否需要重新跑 real-db 测试。

如果一个修复说不清这些，那它大概不是修复，是软件项目的野生蘑菇。不要随便吃。
