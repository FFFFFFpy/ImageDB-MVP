# MVP1 状态与完成标准

## 1. 当前结论

MVP1 当前定性为：

```text
功能完成，进入 Debug / 实战测试阶段。
```

这表示：

- 主功能齐全。
- 本地主链已经人工跑通。
- `main` 已经是 MVP1 基线。
- 后续默认不再扩 MVP1 功能。
- 当前重点转为实战数据、稳定性、安装环境、恢复路径和诊断。

## 2. 已完成主链

MVP1 已完成并通过本地人工验收的主链：

```text
全新开始
→ 初始化托管本地 PostgreSQL
→ 选择源目录
→ 导入 / 分析
→ 审核重复 / 相似候选
→ 人工复核导入计划（draft，可调整导入 / 跳过）
→ 锁定导入计划（frozen，生成 plan hash）
→ 确认并开始入库
→ 本地目录正式入库
```

关键状态口径：

- `review_required`：需要人工审核。
- `ready_to_commit`：已经可以生成并人工复核 draft 计划，不直接跳 Commit。
- Commit 页面只接受已有 frozen / consumed plan 且 `plan_hash` 非空的 run。
- `completed` 不再进入默认 Commit 页。

## 5. Release DoD 口径

当前 release DoD 的正确解释：

- MVP 主链本地人工验收：**通过**。
- 单项测试、Clippy、Release 构建与本地 install-gate：**通过**。
- 完整 clean Windows `pnpm release:gate`：**未签字**。
- 正式 release publication：**未发生**。

所以当前版本不是“未完成”，而是：

```text
MVP1 功能完成
↓
Debug / 实战测试
↓
Release Candidate
↓
正式发布
```

## 6. 后续改动准入规则

### 可以进入 MVP1 Debug 的改动

- 实战测试发现的阻断 bug。
- 数据一致性、文件事务、恢复路径 bug。
- 安装包、PostgreSQL runtime、路径、权限、编码问题。
- 性能瓶颈修复。
- 错误提示和诊断增强。
- 测试和文档补充。

### 默认拒绝的改动

- 新功能。
- 新 UI 大改。
- 新算法方向。
- 非必要 schema 大改。
- 大规模重构。
- 任何没有真实问题支撑的“顺手优化”。

`Debug` 阶段不是把项目重新摊开重做。虽然这句话听起来像废话，但软件行业显然需要反复提醒。
